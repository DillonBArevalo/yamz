Metadictionary guidelines

Metadata and data

Metadata is structured information for describing things we care about. It looks like text with extra punctuation to mark different regions, and elements and values within those regions. That structure makes the information readable by software, which in turn helps us organize, manage, analyze, and discover things we care about.

Real computational power comes when there is wide agreement on metadata terms – on element names and the values they take on. The goal of YAMZ is to support finding, creating, and achieving community accord on high-quality terms. These are the terms that you and others will use to create high-quality metadata. Unlike a regular dictionary, the metadictionary is about terms used to create structured (machine-readable) information.

YAMZ supports data too. Like metadata, data is structured information, but with a shift in emphasis away from being about one particular thing.

Things and assertions about them

The structured information in metadata is usually a group of assertions about a particular thing, such as a document. For example, metadata may assert that a given document's title is X, it's author Y, and it's publication date Z.

Term definitions often refer to the thing being described using a word like "thing", "resource", or "object", eg, "Title: A name given to the resource". Which exact thing that is cannot be found in a term definition. Instead it is usually revealed by metadata values or context (eg, metadata embedded in a file can often be assumed to describe the rest of the file).

To be useful across lots of software and domains, that the elements and value A blob of metadata "record" describing a "thing" consists of one or more statements that themselves may contain smaller statements, all the way down to the finest levels of structure: named elements and their values. Such a record may exist in a digital file or a database, and the described thing might be a the same digital file, a file nearby, or a physical object.

What most concerns the YAMZ metadictionary, however, is coming up with a common vocabulary of named elements, and often their values, from which people can describe things by putting together the best structured statements (metadata records), that is, the ones that are as widely understood as possible. It is aggressively cross-domain and includes all parts of metadata speech -- names, values, encodings, etc. This is not the first online dictionary of metadata; it is Yet Another Metadata Zoo (YAMZ).

The YAMZ Metadictionary is an open registry of metadata terms from all domains and from all parts of "metadata speech". With no login required, anyone can search for and link to term definitions. Register once, then login to create and edit your terms. YAMZ was initially developed in 2013-2014 by a Metadata Working Group within the NSF-funded DataONE initiative, and extended with support from the Digital Library Federation.

The metadictionary aims to be a high-quality cross-domain metadata vocabulary that is directly connected to evolving user needs. Change will be rapid and affordable, with no need for panels of experts to convene and deliberate to improve it. We expect dramatic simplification of vocabulary evolution compared to traditional methods that have created an overwhelming number of vocabularies (ontologies) to choose from.

The hope is that users (people creating machine-readable descriptions of objects they care about) will be able to find most of the terms they need in one place namely, the Metadictionary. One vocabulary, one namespace. This should reduce the number of namespaces, namespace clutter, and the expense of maintaining crosswalks with other vocabularies. And the vocabulary is "shovel-ready" for linked data application, each term having its own permalink (an ARK identifier).

The source code for the YAMZ metadictionary is available on Github as a Python package under a BSD open source license. Full API documentation is also available.

Basic metadictionary structure

Across the dictionary there are three disjoint term classes. Classification is ongoing and fully-automated, based on voting and user reputation.

Vernacular

  • all terms are born here and considered to be works-in-progress
  • unstable terms; permalink points to a definition that may change
  • anyone can propose new vernacular terms
  • proposer(s) of a term "own" it initially; only they can make changes
  • communities of interest spring up around related term clusters
  • term tagging make it easy for communities (eg, working groups) to switch focus to selected term subsets

    • Canonical

  • terms safe for short- or long-term reference
  • stable terms; permalink points to an unchanging definition
  • terms move from vernacular to canonical by consensus
  • terms still subject to voting

    • Deprecated

  • stable terms, but deprecated for long-term reference
  • terms move from canonical to archival by consensus
  • archival terms have links such as "ObsoletedBy: term X"

    • CC0

    All parts of every term, including definitions, examples, and illustrations are dedicated to the public domain under the terms of the CC0 license.
